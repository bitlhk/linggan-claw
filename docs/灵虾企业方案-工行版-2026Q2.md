# 二、业务逻辑：人与业务解耦

## 核心设计：Agent 既可代表人，也可代表业务

```
                ┌────────────────────────────┐
                │       智能体广场            │
                │  （Agent Marketplace）      │
                └─────────────┬──────────────┘
                              │
        ┌─────────────────────┼─────────────────────┐
        │                     │                     │
    👤 人类型 Agent         📋 业务型 Agent       🔧 工具型 Agent
    （代表某个员工）        （代表某项业务能力）  （代表某个技能集）
        │                     │                     │
   小张的灵虾              金融投顾                代码助手
   小王的灵虾              股票分析                幻灯片生成
   领导的灵虾              年报解读                数据分析
        │                     │                     │
        └──── 互相协作 ───────┴────────────────────┘
              （审批 + 执行 + 回应）
```

**关键理念**：
- 人和业务**解耦** —— 业务 Agent 由企业统一维护（如风控、投顾），员工 Agent 是个人助手
- 人 Agent 调用业务 Agent，就像下属向部门提需求
- 业务 Agent 可被全公司复用，不重复造轮子

## 多 Agent 协作流程

```
小张的虾                                       王行长的虾
    │                                              │
    │   ① 帮我让王行长审批一份年报分析报告           │
    │                                              │
    ├──── 发起协作请求 ─────────────────────────►   │
    │                                              │
    │                                              │  ② 弹出协作请求通知
    │                                              │     "小张请求您审阅年报"
    │                                              │
    │                                              ├──┐
    │                                              │  │ ③ 王行长 / 王行长的虾
    │                                              │  │   人工审批 / 自动审批
    │                                              │◄─┘
    │                                              │
    │   ④ 审批通过 + 反馈意见                       │
    │◄──────────────────────────────────────────── │
    │                                              │
    │   ⑤ 继续后续工作                              │
```

**精髓**：
- **发起方**（小张）不需要知道对方的 prompt 是什么
- **被调用方**（王行长）可以**选择性回复** —— "同意/拒绝/修改后回复/转交他人"
- **不改提示词** —— 协作不污染各自的人格设定
- **审批权在人，执行权在 Agent** —— 决策和动作分离

## 关键设计：为什么是「一个 Gateway · 多个 Agent」？

### 两种架构对比

| 维度 | ❌ 每人一个 Gateway | ✅ 一 Gateway 多 Agent（灵虾选择） |
|---|---|---|
| **资源占用** | 每用户独占进程，1000 用户 = 1000 进程 | 单进程多 session，1000 用户 ≈ 1 进程 |
| **冷启动** | 每次启动 10-30 秒 | 即时切换 |
| **内存成本** | 100MB × N | 500MB 总量 |
| **协作能力** | 跨进程通信复杂 | 同进程内消息直传 |
| **故障半径** | 一个挂全员断 | session 隔离，单 session 异常不影响他人 |
| **运维复杂度** | N 套监控告警 | 一套告警 |
| **K8s 友好度** | 难，状态太散 | 好，无状态扩缩 |

### Session 隔离原理

```
                  Gateway 进程
                 ┌──────────────┐
   小张的虾 ───► │ session A    │ ──► 工作区 A / 记忆 A / 技能 A
                 │              │
   小王的虾 ───► │ session B    │ ──► 工作区 B / 记忆 B / 技能 B
                 │              │
   小李的虾 ───► │ session C    │ ──► 工作区 C / 记忆 C / 技能 C
                 └──────────────┘

   每个 session 有独立的 key（agent:trial_lgc-{UUID}:main）
   工作区物理隔离（/root/.openclaw/workspace-{agentId}/）
   工具调用走独立的沙箱容器
```

## 多端访问与上下文同步

```
   办公室电脑                家里电脑                  手机
       │                        │                       │
       └────────┬───────────────┴───────────────────────┘
                │
                ▼
           灵虾后端
                │
                ▼
       ┌────────────────┐
       │ Gateway        │
       │  Session Key   │
       │  agent:xxx:main│  ← 唯一确定性 key
       └────────────────┘
                │

        同一份上下文

```

**WebSocket 走不通就回退 HTTP SSE**，两条路径打到**同一个 session key**，企业内网防火墙友好。

# 四、安全保障：金融级护栏

## 双层安全：OpenClaw 内核 + 灵虾平台

```
┌────────────────────────────────────────────────────────┐
│                                                        │
│  灵虾平台层（业务安全 / 合规审计 / 权限治理）          │
│                                                        │
│  • 用户认证（OAuth2 + 设备指纹）                       │
│  • IP 访问限流（每 IP 每日配额）                       │
│  • 操作审计日志（每条 API 调用全量记录）               │
│  • 权限分级（starter / plus / internal）               │
│  • 文件下载 token 短期签名（HMAC-SHA256）              │
│  • 协作请求审批流                                      │
│                                                        │
├────────────────────────────────────────────────────────┤
│                                                        │
│  OpenClaw 内核层（运行时安全 / 沙箱隔离 / 工具治理）   │
│                                                        │
│  • Docker 沙箱（network=none + read-only）             │
│  • 资源配额（CPU/内存/PID 限制）                       │
│  • 命令黑名单（sudo/mount/dd 等阻断）                  │
│  • 三级审批（高危命令需人工批准）                      │
│  • 工作区隔离（per-agent 物理隔离）                    │
│  • 工具调用审计                                        │
│                                                        │
└────────────────────────────────────────────────────────┘
```

## 沙箱安全配置（每次工具执行）

| 安全项 | 配置 | 作用 |
|---|---|---|
| 网络隔离 | `--network none` | 沙箱无外网，防数据外泄 |
| 文件系统 | `--read-only` | 根目录只读，防恶意写入 |
| 临时空间 | `--tmpfs /tmp:size=50m` | 限制临时文件大小 |
| 内存限制 | `--memory 256m` | 防 OOM 攻击 |
| CPU 限制 | `--cpus 0.5` | 防计算资源耗尽 |
| 进程数 | `--pids-limit 50` | 防 fork 炸弹 |
| Linux Cap | `--cap-drop ALL` | 移除所有特权 |
| 提权 | `--security-opt no-new-privileges` | 禁止 setuid 提权 |
| 超时 | 默认 10s，最长 30s | 防死循环 |
| 输出限制 | 64KB | 防输出爆炸 |

## 命令黑名单（运行时拦截）

```
sudo / su          → 提权拦截
chmod +s           → setuid 拦截
mount / unshare    → 容器逃逸拦截
dd /dev/*          → 块设备写入拦截
nsenter            → 命名空间穿透拦截
/proc/sysrq        → 内核操作拦截
```

## 权限分级（按企业角色）

| 等级 | 适用人群 | 能力 |
|---|---|---|
| **Starter** | 普通员工试用 | 基础对话、只读工具 |
| **Plus** | 正式员工 | 沙箱执行、技能安装、协作 |
| **Internal** | 内部研发 | 全部能力 + 调试接口 |

## 安全协同矩阵

| 安全维度 | OpenClaw 内核 | 灵虾平台 | 协同效果 |
|---|---|---|---|
| **身份认证** | Gateway Token | OAuth2 + Cookie | 双重认证 |
| **会话隔离** | session key | adoptId 绑定 | 物理 + 逻辑双隔离 |
| **数据加密** | 工作区文件系统 | 数据库字段加密 | 端到端 |
| **网络隔离** | 沙箱 network=none | 外网通过白名单代理 | 全栈隔离 |
| **审计追溯** | 工具调用日志 | tRPC 全量审计日志 | 完整审计链路 |
| **权限控制** | 三级审批 | 权限配置文件 | 双重把关 |
| **限流防爆** | 沙箱并发限制 | IP/用户级限流 | 多层防御 |

# 五、可靠性方案：从单机到 K8s

## 故障点分析与应对

| 故障点 | 影响范围 | 现状应对 | 演进方向 |
|---|---|---|---|
| **OpenClaw Gateway 进程崩溃** | 全员断线 | PM2 自动重启（< 5s） | Keepalived 主备 + K8s 探针 |
| **WebSocket 空闲断连** | 长任务输出丢失 | **30 秒心跳保活**（已实现） | 同 |
| **WS 重连上下文丢失** | 用户感知"失忆" | **复用 main session key**（已实现） | 同 |
| **数据库连接异常** | 读写失败 | 自动重连 + 慢查询监控 | 读写分离 + 哨兵 |
| **沙箱 OOM / 死循环** | 单次任务失败 | 资源配额 + 超时强制 kill | 同 |
| **模型 API 限流** | 部分请求失败 | 多模型 fallback | 智能路由 + 本地模型保底 |
| **磁盘空间耗尽** | 工作区写入失败 | 定时清理 + 告警 | 自动清理 + 配额管理 |
| **WSS 端口被企业防火墙拦截** | 聊天不可用 | **HTTP SSE 自动回退** | 同 |

## 5 分钟 Demo 流程（技术人版）

```
00:00 - 01:00   申领 + 架构透传
                操作："看，30 秒申领一只虾，零安装，能用了"
                技术讲解：
                  · 底层是 OpenClaw 开源 gateway，每个用户独立 agent session + workspace
                  · per-user 架构，不是中心化编排——每只虾有自己的记忆、技能、人格、沙箱
                  · 支持多模型路由：DeepSeek / GLM / Claude，按场景切
                预判提问：
                  Q: "底层是什么 runtime？"
                  A: "OpenClaw gateway + Docker 沙箱，每个用户独立 session"
                  Q: "跟你们自己的模型绑定吗？"
                  A: "不绑定，多模型路由，企业可以接自己的模型"

01:00 - 02:00   技能市场 = 企业内部 AI 应用商店
                操作：在对话里安装一个"股票分析"技能
                技术讲解：
                  · 技能 = system prompt + tools 组合包，用户自助创建、可发布到市场
                  · 相当于企业内部 AI 应用商店——每个部门自己发布技能，不用排 IT 需求
                  · 技能跑在用户自己的虾里，workspace 天然隔离，无跨用户数据泄漏
                预判提问：
                  Q: "技能跟 function calling 什么关系？"
                  A: "技能包含 function calling 但不止于此，还有 system prompt + 上下文模板 + 工具组合"

02:00 - 03:00   业务 Agent + 工具执行安全
                操作：打开灵犀·股票分析（TaskPanel 里点图标直接用）
                技术讲解：
                  · 业务 Agent 是 DB 显式注册的，有独立 api_url / auth token / 权限配置
                  · 工具执行走 Docker 沙箱：--network none / --read-only / cap-drop ALL / tmpfs / pids-limit
                  · 每次工具调用都有审计落库：谁调了什么、参数、结果、耗时
                操作：让虾执行一段代码 → 展示审计日志
                预判提问：
                  Q: "代码执行安全怎么做？"
                  A: 展示 Docker 参数清单 + tool_router 三层策略
                  Q: "命令策略是什么模型？"
                  A: "白名单 + 黑名单 + 参数校验。威胁模型是受信员工 + agent 幻觉防护。
                      需要零信任级别时可上强语义策略引擎 + eBPF 监控"

03:00 - 04:00   多租户隔离 + 协作审批
                操作：展示两个账号的 workspace 互相不可见
                技术讲解：
                  · TIL（Tenant Isolation Layer）：userId x agentId → tenantToken → 独立 workspace
                  · 跨用户协作走审批流：A 发 request → B 审批 → 自动执行 → 全程可追溯
                操作：发起协作请求 → 切账号审批 → 展示审计记录
                预判提问：
                  Q: "隔离到什么粒度？"
                  A: "user x agent 粒度，同一用户不同业务 agent 也互相隔离"

04:00 - 05:00   安全护栏 + 总结
                操作：尝试 sudo / 尝试访问外网 → 展示拦截
                技术讲解：
                  · 五层防线：Docker 沙箱 → tool_router 白名单 → 参数校验 → 租户 TIL → 审计落库
                总结：
                  "我们做的不是又一个套壳产品，是一个可定制、可隔离、可协作、可治理的企业 Agent 工作台"

                预判提问：
                  Q: "开源吗？"
                  A: "核心平台开源，品牌/多租户/审计等企业特性走商业授权"
                  Q: "工行内部已有 XXX 平台了，你跟它有什么区别？"
                  A: 不正面比较，走互补叙事——
```

## 深水区 Q&A 预案（演示后自由讨论）

| 问题 | 回答要点 |
|---|---|
| 怎么处理 prompt injection？ | 五层防线 + 威胁模型是受信员工+幻觉防护。需要零信任级别时上强语义策略引擎 |
| 多模型路由怎么做的？ | 用户级可配 + 管理员可设默认。支持 DeepSeek / GLM / Claude，加新模型只需适配 adapter |
| 跟 Dify / Coze / 百炼 有什么区别？ | 那些是"给开发者的 agent 构建工具"，灵虾是"给企业员工的 agent 使用平台"。他们解决"怎么造 agent"，我们解决"怎么让 1000 个员工每人有一个 agent 且可管控" |
| 数据合规 / 私有化部署？ | 全链路数据不出客户网络；可接客户自有大模型；审计日志可对接 SIEM |
| 团队多大？能支撑企业级？ | 坦诚小团队。核心平台开源 + 社区驱动 + 企业版走定制。参考 GitLab 早期模式 |
| 金融 agent 回答不够专业？（如果秀了 Hermes） | "这是通用业务理解力展示。上线后会把贵行的业务规则、合规要求接入 agent context，给出定制化答案" |
