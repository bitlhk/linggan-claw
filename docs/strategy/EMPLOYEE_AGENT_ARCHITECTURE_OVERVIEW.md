# 员工智能体系统架构总览

**版本**: 2.0 · 面向领导/客户层汇报版
**日期**: 2026-05-11
**性质**: 产品定位 + 系统架构 + 企业部署说明

---

## 一句话定位

员工智能体是面向企业内部员工的 **AI Agent Client**：它把登录、身份、权限、组织协作、技能、任务、渠道和审计封装在一个统一入口之上，让员工可以创建并使用自己的 AI 智能体，同时让企业保留可管理、可隔离、可审计、可部署的控制边界。

它不是单纯的 OpenClaw 包装页，也不是只面向金融行业的任务平台；它更像企业内部的“员工 AI 工作台”，底层可以接 OpenClaw、Hermes、Claude Code、Codex、MCP/A2A 服务或企业自有模型运行时。

## 一、产品边界

```text
灵感平台
  └─ 员工智能体
       ├─ 员工个人智能体
       ├─ 智能体工作台
       ├─ 智能体广场
       ├─ 技能广场
       ├─ 任务工作台
       ├─ 组织协作
       └─ 管理后台

底层运行时
  ├─ OpenClaw: 通用对话、工具调用、文件工作区、技能执行
  ├─ Hermes: 专业 Agent / Financial Agent Harness / 多 worker 编排
  ├─ Claude Code / Codex: 代码、文档、PPT/Office 产物生成
  └─ MCP / A2A / 搜索服务: 外部工具、数据源、远端 Agent
```

**边界说明**：

- **灵感** 是用户登录和 AI 应用入口。
- **员工智能体** 是企业员工可使用、可管理、可协作的 Agent Client。
- **OpenClaw / Hermes / Claude Code / Codex** 是运行时底座，不直接作为最终产品名面向用户。
- **技能广场 / 任务工作台 / 智能体广场** 是员工智能体内的能力组织方式。

## 五、用户隔离与会话隔离

员工智能体面向企业部署时，隔离策略按“人、智能体、渠道、会话”四个维度处理。

```text
sessionKey = agent:{runtimeAgentId}:{channel}:{conversationId}

示例:
  Web 主会话      agent:trial_lgc_xxx:web:main
  Web 新对话      agent:trial_lgc_xxx:web:conv_123
  微信对话        agent:trial_lgc_xxx:wechat:openid_xxx
  飞书对话        agent:trial_lgc_xxx:feishu:user_xxx
  协作空间对话    agent:trial_lgc_xxx:coop:space_task_xxx
```

这样可以同时满足两件事：

- 同一员工在同一网页会话中保持上下文连续。
- 网页、微信、飞书、协作空间之间不串上下文。

同时，后端按 `adoptId / runtimeAgentId / userId / spaceId` 做权限校验，避免一个员工看到或操作另一个员工的智能体、定时任务、技能和协作内容。

## 六、安全与合规边界

```text
┌─────────────────────────────────────────────────────────────────┐
│ 安全边界                                                         │
├─────────────────────────────────────────────────────────────────┤
│ 身份边界: 登录态、管理员/员工权限、组织空间成员关系              │
│ 智能体边界: 每个员工智能体独立 runtimeAgentId 和工作区            │
│ 会话边界: channel + conversationId 隔离上下文                    │
│ 工具边界: OpenClaw/Hermes 工具白名单、写入者约束、MCP 权限声明   │
│ 数据边界: 私有文件、技能、记忆、定时任务按用户和智能体隔离        │
│ 审计边界: 创建、调用、发布、审核、删除、推送均可留痕              │
└─────────────────────────────────────────────────────────────────┘
```

当前策略强调“企业可控”：默认关闭业务 Agent 预置、任务工作台灰度开启、外部搜索/MCP 能力显式配置、管理员后台统一管理用户和智能体能力。

## 七、部署模式

### 单机裸机部署

适合试点、PoC、小团队内部验证。

```text
企业服务器
  ├─ 员工智能体 Node 服务 (:5180)
  ├─ MySQL
  ├─ OpenClaw 本机运行时
  ├─ 可选 Hermes / Claude Code / Codex
  └─ PM2 进程管理
```

特点：部署简单，方便用一键脚本安装和升级；默认员工智能体与 OpenClaw 同机，降低网络和鉴权复杂度。

### 企业级扩展部署

适合正式生产。

```text
统一入口 / 反向代理
  ├─ 员工智能体服务集群
  ├─ 独立 MySQL / 备份
  ├─ OpenClaw Runtime 节点池
  ├─ Hermes / 专业 Agent 节点
  ├─ MCP / A2A 服务
  └─ 企业身份源 / 审计平台 / 文件存储
```

特点：运行时可以分层扩展，员工智能体继续承担企业治理入口，不直接绑定单一 AI 底座。

## 八、与旧“灵虾”定位的差异

| 维度 | 旧定位：灵虾 | 当前定位：员工智能体 |
|---|---|---|
| 产品名称 | 灵虾 / 子虾 / 领养 | 员工智能体 / 创建智能体 / 智能体工作台 |
| 行业范围 | 偏银行/金融任务交付 | 面向企业员工的通用 Agent Client |
| 核心入口 | 任务交付平台 | 员工个人 AI 工作台 |
| 技术关系 | 灵虾包装 OpenClaw/Hermes | 员工智能体治理多种运行时 |
| 部署目标 | 金融客户场景验证 | 企业可移植部署与内部推广 |
| 能力组织 | 金融 Agent、技能、定时 | 智能体、技能、任务、渠道、协作、管理 |

旧“灵虾”思路仍然有价值，尤其是金融任务、合规审计、Financial Agent Harness；但对外汇报时，应统一使用“员工智能体”作为产品口径，把金融能力作为可选行业场景，而不是产品本身的唯一定位。

## 九、阶段性结论

员工智能体当前最重要的价值不是“大而全的平台”，而是把企业员工真正会用到的 AI Agent 能力收敛到一个可管理入口：

- 员工能创建和使用自己的智能体。
- 企业能管理用户、权限、技能、任务和渠道。
- 运行时能从 OpenClaw 扩展到 Hermes、Claude Code、Codex、MCP 和 A2A。
- 数据、会话、协作和定时任务要做到隔离。
- 关键任务要可审计、可追溯、可交付。

这也是后续从单机试点走向企业部署时应坚持的主线。
