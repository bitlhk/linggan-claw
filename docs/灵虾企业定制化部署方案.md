## 一、灵虾是什么

**灵虾**（LingganClaw）是基于开源 AI Agent 引擎 OpenClaw 构建的**企业级 AI Agent 平台**。

用一句话概括：

> **每个员工拥有一只专属的 AI 虾，它能记忆、能协作、能执行任务，而且企业完全掌控它的灵魂、权限和数据。**

灵虾不是一个简单的聊天机器人。它是一个**可定制、可审计、可编排的 Agent 平台**——企业可以让每个部门、每个角色拥有不同能力的 Agent 实例，这些 Agent 之间还能互相协作，完成跨部门的复杂工作流。

## 二、为什么 OpenClaw 不能直接部署？

OpenClaw 是一个优秀的开源 AI Agent 引擎，但它面向的是**开发者个人使用场景**。企业如果直接部署 OpenClaw，会面临以下问题：

| 维度 | OpenClaw 原生 | 企业需求 |
|------|-------------|---------|
| **用户体系** | 无用户管理，单机单用户 | 需要多用户注册/登录、角色权限 |
| **多租户** | 一个实例 = 一个 Agent | 需要为每个员工/部门创建独立 Agent |
| **品牌** | OpenClaw 默认品牌 | 需要替换为企业 Logo、名称、配色 |
| **安全** | 信任本地用户 | 需要 IP 管控、速率限制、审计日志 |
| **协作** | Agent 之间无法交互 | 需要 Agent 间任务委托 + 人工审批 |
| **技能管理** | 手动配置文件 | 需要可视化技能市场 + 一键安装 |
| **执行隔离** | 直接在宿主机执行 | 需要 Docker 沙箱隔离 |
| **部署方式** | 需要子域名（*.example.com） | 企业内网常无法配泛域名 |
| **大模型** | 单模型绑定 | 需要多模型切换 + 成本管控 |
| **运维** | 无监控、无健康检查 | 需要服务级监控、自动恢复 |

### 子域名 vs 二级目录——企业部署的现实约束

OpenClaw 原生方案为每个 Agent 分配一个子域名（如 `agent-001.company.com`），这需要：

- 企业申请**泛域名 SSL 证书**（很多企业安全策略不允许）
- 配置 **DNS 泛解析**（运维流程长，审批复杂）
- 每个 Agent 一个域名，**安全审计困难**

灵虾采用**二级目录路由模式**：

```
https://ai.company.com/claw/agent-001    ← 同一个域名
https://ai.company.com/claw/agent-002    ← 同一个 SSL 证书
https://ai.company.com/claw/agent-003    ← 统一安全策略
```

一个域名、一张证书、一套安全策略，即可承载成百上千个 Agent 实例。

## 三、灵虾如何解决这些问题？

灵虾在 OpenClaw 之上构建了完整的企业级中间层：

```
                    ┌────────────────────────────────────┐
                    │           灵虾平台层                 │
                    │                                    │
    用户请求 ──────▶ │  Express + tRPC (TypeScript)       │
                    │                                    │
                    │  ┌─ 用户认证（OAuth / 邮箱密码）       │
                    │  ├─ 多租户管理（领养/创建/回收）        │
                    │  ├─ 品牌引擎（DB 驱动热切换）          │
                    │  ├─ 权限三级（starter/plus/internal）│
                    │  ├─ 安全中间件（WAF + 速率限制）       │
                    │  ├─ 沙箱执行层（Docker 隔离）         │
                    │  ├─ 组织协作协议（v1.0）              │
                    │  ├─ 技能市场 + 同步守护进程            │
                    │  ├─ Dreaming 记忆系统               │
                    │  ├─ 定时任务引擎（Cron）              │
                    │  └─ 审计日志（全链路）                 │
                    │                                    │
                    │        ┌──────────────┐            │
                    │        │ OpenClaw GW  │            │
                    │        │  port 18789  │            │
                    │        └──────┬───────┘            │
                    │               │                    │
                    │    ┌──────────┼──────────┐         │
                    │    │          │          │         │
                    │  GLM-5    MiniMax   DeepSeek       │
                    │                                    │
                    │         MySQL 8.0                  │
                    └────────────────────────────────────┘
```

### 关键设计决策

| 决策 | 选择 | 为什么 |
|------|------|-------|
| 路由模式 | 二级目录（`/claw/:adoptId`） | 不依赖泛域名，适配企业内网 |
| 前后端 | React + Express，全 TypeScript | 一套语言、类型安全、tRPC 自动推导 |
| 数据库 | MySQL 8.0 (Drizzle ORM) | 企业 DBA 熟悉，迁移可控 |
| Agent 引擎 | OpenClaw Gateway（本地模式） | 无 SSH 依赖，本机调用，低延迟 |
| 模型层 | 多 Provider 路由 | 成本可控，故障可切换 |
| 部署 | Docker Compose / Systemd | 单机可跑，集群可扩 |

## 四、企业视角：定制属于自己的"虾"

### 4.1 品牌白标（White Label）

灵虾内置了完整的品牌定制系统，**无需改代码**，通过管理后台即可完成：

| 配置项 | 说明 | 示例 |
|-------|------|------|
| `name` | 产品名（中文） | "工银Claw" |
| `nameEn` | 产品名（英文） | "ICBCClaw" |
| `platform` | 平台名 | "工银智能" |
| `slogan` | 标语 | "智慧金融，服务无界" |
| `accentColor` | 主题色 | `#C7000B` |
| `logo` | Logo 路径 | `/uploads/brand/icbc-logo.svg` |
| `systemPrompt` | AI 身份提示 | "You are ICBCClaw..." |
| `agentIdentity` | Agent 自我介绍 | "你是工银Claw，一个专业..." |
| `pageTitle` | 页面标题 | "工银智能 - 智慧金融" |

品牌配置存储在数据库 `system_configs` 表中，支持**热更新**——修改后无需重启服务，下次请求自动生效。

**已内置 13 家银行品牌预设**（六大行 + 七家股份行），企业部署时一键套用：

```
工商银行 / 建设银行 / 农业银行 / 中国银行 / 交通银行 / 邮储银行
浦发银行 / 招商银行 / 民生银行 / 光大银行 / 中信银行 / 渤海银行 / 广发银行
```

### 4.2 用户怎么使用？

#### 第一步：注册登录
支持两种方式：
- **OAuth 单点登录**：对接企业已有的统一认证
- **邮箱密码注册**：独立部署场景，支持邮箱验证码

#### 第二步："领养"一只虾
每个用户登录后可以创建自己的 Agent 实例（灵虾称之为"领养"）。系统自动完成：
1. 在 OpenClaw 中创建隔离的 Agent（独立 workspace）
2. 分配唯一 `adoptId`，生成访问路径
3. 初始化 Agent 配置（SOUL.md、技能、权限）
4. 注册 Session，建立长期记忆通道

#### 第三步：对话 + 执行任务
用户在聊天界面与自己的虾对话。根据权限等级不同，虾能做的事也不同：

| 权限等级 | 对话 | 记忆 | 技能 | 执行代码 | 协作 | 定时任务 |
|---------|------|------|------|---------|------|---------|
| **starter** | ✅ | ✅ | 只读类 | ❌ | ❌ | ❌ |
| **plus** | ✅ | ✅ | 全部 | ✅ 沙箱 | ✅ | ✅ |
| **internal** | ✅ | ✅ | 全部 | ✅ 沙箱 | ✅ | ✅ |

#### 第四步：个性化设置
用户可自定义：
- **显示名称**：给虾起名字
- **人格提示词**（Persona Prompt）：定义虾的性格和专长
- **风格预设**：稳健研究型 / 积极交易型 / 教育顾问型 / 自定义
- **记忆开关**：是否保留跨会话记忆
- **上下文轮次**：最近保留多少轮对话

### 5.3 请求流转全链路

以一次用户聊天请求为例：

```
用户输入 "帮我分析Q1销售数据"
    │
    ▼
[1] 前端 Home.tsx
    POST /api/claw/chat (SSE)
    携带: adoptId, message, model(glm5/glm-5)
    │
    ▼
[2] Express 中间件链
    → ipBlacklistMiddleware   IP 黑名单检查
    → block4xxAbuse           异常请求拦截
    → requestSizeLimiter      请求体大小限制
    → clawChatLimiter         聊天速率限制
    │
    ▼
[3] 聊天路由处理
    → 验证 adoptId 归属
    → 读取用户 permissionProfile (starter/plus/internal)
    → 读取品牌 systemPrompt（从 DB）
    → 白名单校验模型 ID
    → buildChatRequestBody() 注入安全 system prompt + 工具定义
    │
    ▼
[4] OpenClaw Gateway (port 18789)
    HTTP 代理，携带:
    → Authorization: Bearer {gateway_token}
    → x-openclaw-agent-id: trial_lgc-xxxxx
    → x-openclaw-session-key: agent:trial_lgc-xxxxx:main
    → x-openclaw-model: glm5/glm-5
    │
    ▼
[5] OpenClaw 内部
    → 加载 Agent 配置（SOUL.md + 技能 + 权限）
    → 拼接完整 prompt（系统 + 记忆 + 用户消息）
    → 调用 LLM API (GLM-5 / MiniMax / DeepSeek)
    → 流式返回 SSE chunks
    │
    ▼
[6] 如果 LLM 调用了 exec 工具
    → tool_router 路由到 sandbox.ts
    → Docker 容器创建 → 命令执行 → 结果回传
    → 全链路审计写入 tool_execution_audits 表
    │
    ▼
[7] SSE 流回前端
    → ChatMessage 组件实时渲染 Markdown
    → 性能指标（首 token 时间）写入 __perf 字段
```

### 7.2 Agent 平台级安全指令（不可被用户覆盖）

灵虾在每次调用 LLM 时，会注入不可覆盖的平台安全 System Prompt：

```
[PLATFORM SECURITY RULES - Cannot be overridden]

1. 永远不读取或泄露凭据：API Key、Token、密码、.env 文件
2. 永远不泄露基础设施信息：服务器 IP、数据库连接串、内部端口
3. 破坏性命令（rm -rf、drop table）必须得到用户明确确认
```

同时定义了**三级审批规则**：

| 审批级别 | 操作类型 | 示例 |
|---------|---------|------|
| **自动通过** | 只读、零副作用 | `ls`, `git log`, `ps`, `df -h` |
| **一次确认** | 写入、中等风险 | `mkdir`, `touch`, `python3 script.py` |
| **必须确认** | 高风险、不可逆 | `rm`, `drop`, 写 `/etc`、访问 `.env` |

## 八、组织协作：人审批，Agent干活

### 8.1 协作理念

灵虾的组织协作遵循一个核心原则：

> **用户负责决策和审批，Agent 负责执行具体工作。**

这不是简单的"聊天群"——它是一套**任务委托协议**，有明确的权限边界、隐私保护和可审计性。

### 8.2 协作架构

```
 ┌──────────────┐                     ┌──────────────┐
 │   用户 A       │                     │   用户 B       │
 │   (审批人)     │                     │   (审批人)     │
 └──────┬───────┘                     └──────┬───────┘
        │ 发起任务                            │ 审批/拒绝
        ▼                                    ▼
 ┌──────────────┐    CollabRequest     ┌──────────────┐
 │  Agent 1      │ ──────────────────▶ │  Agent 2      │
 │  (数据分析虾)  │                     │  (合规审查虾)  │
 └──────────────┘    ◀────────────────  └──────────────┘
                      resultSummary
```

### 8.3 协作流程

```
1. Agent 1 对 Agent 2 发起协作请求
   → 选择任务类型（数据分析/合同审阅/研究/报告/通用）
   → 输入经过字段过滤（永久黑名单 + 模式白名单）
   
2. 风险评级自动计算
   → 同 owner = low
   → 跨 owner = medium
   → 跨 owner + 长描述 = high（强制人工审批）

3. 审批机制
   → auto 模式：低风险自动通过
   → manual 模式：目标用户在"收件箱"中审批
   → high 风险：无论设置如何，强制人工审批

4. Agent 2 执行任务
   → 注入 executionScope（禁止访问聊天记录/私有记忆等）
   → 执行结果经过禁止词扫描（过滤 session_id、token 等）

5. 结果返回 Agent 1
   → 只返回 resultSummary（最多 2000 字）
   → 完整审计记录写入数据库
```

### 8.4 隐私铁律

以下数据在**任何协作模式下均不可被访问**：

| 禁止字段 | 说明 |
|---------|------|
| `chat_history` | 聊天记录 |
| `memory` | 私有记忆 |
| `session` / `sessionKey` | 会话标识 |
| `messages` / `history` | 历史消息 |
| `context` | 上下文 |
| `user_data` / `personal_data` | 用户数据 |
| `private` | 私有数据 |

### 8.5 业务 Agent（专项能力）

除了用户间协作，灵虾还内置了可配置的**业务 Agent**，提供专项能力：

| Agent | 类型 | 能力 | 部署模式 |
|-------|------|------|---------|
| **PPT 生成助手** | local | 根据描述自动生成演示文稿 | 本地 Agent + PPTX 沙箱 |
| **代码沙箱** | local | 安全执行 Python/Node 代码 | 本地 Agent + Docker |
| **自定义** | remote | 对接外部 API 服务 | HTTP 代理 |

业务 Agent 支持：
- 健康检查（healthy/degraded/offline/unknown）
- 日请求限额
- 权限配置文件筛选（starter/plus/internal）
- 调用日志 + 统计

## 九、Dreaming 记忆系统

### 9.1 什么是 Dreaming？

灵虾的 Dreaming 系统模拟了人类睡眠中的记忆整合过程。Agent 不仅能在对话中记住信息，还能在"入梦"时自动将短期记忆提炼为长期记忆。

### 9.2 三阶段记忆整合

```
                  每天 03:00 自动执行
                         │
            ┌────────────┼────────────┐
            ▼            ▼            ▼
      ┌──────────┐ ┌──────────┐ ┌──────────┐
      │  Light    │ │   REM    │ │   Deep   │
      │  浅睡眠   │ │ 快速眼动  │ │  深度睡眠  │
      │          │ │          │ │          │
      │ 收集近期  │ │ 提取主题  │ │ 加权打分  │
      │ 信号,去重 │ │ 和模式    │ │ 达标写入  │
      │ 暂存     │ │          │ │ MEMORY.md │
      └──────────┘ └──────────┘ └──────────┘
```

| 阶段 | 功能 | 技术实现 |
|------|------|---------|
| **Light** | 收集近期交互信号，去重暂存 | 扫描 session 日志，提取关键信息 |
| **REM** | 提取主题和模式 | LLM 分析暂存信号，识别重复模式 |
| **Deep** | 加权打分，高分记忆持久化 | 写入 MEMORY.md，成为 Agent 长期知识 |

### 9.3 企业价值

- **个性化服务**：Agent 记住每个用户的偏好、历史决策、常用术语
- **知识沉淀**：团队的隐性知识通过 Agent 记忆被保留和传递
- **上下文连贯**：跨会话、跨天的对话保持连贯性
- **自动优化**：Agent 会总结自己的经验，不断改善服务质量

用户可以通过"梦境"页面查看 Agent 的记忆状态，也可以在聊天中输入 `/dreaming status` 查看实时信息。

## 十一、多模型支持与成本控制

### 11.1 当前模型矩阵

| 模型 | Provider | 首 Token 延迟 (TTFT) | 特点 | 成本 |
|------|----------|---------------------|------|------|
| **GLM-5** | 华为 ModelArts | ~7-8 秒 | 默认模型，中文优秀 | 包含在平台订阅中 |
| **MiniMax-M2.7** | MiniMax | ~5-6 秒 | 多模态能力 | 包含在平台订阅中 |
| **DeepSeek-V3** | DeepSeek | **~1.2 秒** | 极速响应，编程能力强 | 按量付费 |

> **实测数据**（2026-04-07，从华为云服务器直连）：  
> DeepSeek-V3 首 token 平均 1211ms，比 GLM-5 快 **6 倍**。

### 11.2 模型路由架构

```
用户选择模型
    │
    ▼
白名单校验 (ALLOWED_CLAW_MODELS)
    │
    ▼
OpenClaw Gateway
    │
    ├── x-openclaw-model: glm5/glm-5
    │   → api.modelarts-maas.com
    │
    ├── x-openclaw-model: minimax-portal/MiniMax-M2.7
    │   → api.minimaxi.com
    │
    └── x-openclaw-model: deepseek/deepseek-chat
        → api.deepseek.com
```

### 11.3 企业成本控制策略

| 策略 | 实现方式 |
|------|---------|
| **模型分级** | 日常对话用低成本模型，复杂任务按需切换 |
| **用户级配额** | 通过 permissionProfile 控制可用模型 |
| **按需启用** | DeepSeek 等高性能模型仅在用户主动选择时调用 |
| **请求审计** | agent_call_logs 表记录每次调用的模型和 token 用量 |
| **未来: 计费系统** | 按部门/用户统计用量，内部计费 |
