# Enterprise Audit Ledger v1

Enterprise Audit Ledger is the neutral audit baseline for the platform. It is designed for financial-customer deployments and future runtime types such as employee, Hermes, Jiuwen, and custom Claw runtimes.

## Goals

- Provide one audit entry point for auth, admin, agent lifecycle, channel, model, skill, tool, browser, file, tenant, security, and system events.
- Use neutral names without legacy product prefixes.
- Keep audit events append-only by default.
- Support export, filtering, retention planning, and future SIEM forwarding.
- Keep legacy audit tables out of the new primary audit path.

## Tables

### `audit_events`

The immutable audit ledger. All important actions write here.

Key fields:

- `event_id`: external ID, generated by the application with ULID. DB `id` is internal only.
- `event_time`: passed explicitly by the application. Do not depend on DB server time as the single source.
- `category` and `action`: dot notation such as `auth.login.success`.
- `actor_*`, `target_*`, `resource_*`: separate the person, object, and affected resource.
- `workspace_id`, `agent_instance_id`, `runtime_type`, `runtime_agent_id`: connect events to runtime topology.
- `request_id`, `session_id`, `correlation_id`: trace single requests and batch operations.
- `error_code`, `policy_code`, `risk_type`, `channel`, `tool_name`: nullable hot columns for indexed queries.
- `metadata_json`: limited to 16 KB after redaction and truncation.

### `audit_tool_events`

Tool, shell, browser, MCP, file, and HTTP execution detail. It is keyed by `event_id` and points back to `audit_events`.

### `audit_security_findings`

Security findings and their handling state. Unlike other audit tables, this table permits controlled updates to:

- `status`
- `handled_by`
- `handled_at`
- `handled_note`
- `updated_at`

All other fields are immutable.

Valid status transitions:

- `open -> acknowledged`
- `open -> resolved`
- `open -> ignored`
- `acknowledged -> resolved`
- `acknowledged -> ignored`

Resolved and ignored findings do not reopen in v1.

### `audit_exports`

Tracks audit exports. Export actions are themselves audited.

Use `storage_key`, not a physical file path. Download must go through a controlled admin endpoint that verifies permission, expiry, and file hash.

## WORM

WORM is enforced at two layers:

1. Database triggers reject UPDATE/DELETE on immutable audit tables.
2. Runtime DB privileges keep the application account from changing schema, triggers, or existing rows.

Runtime application account:

- `SELECT`, `INSERT` on `audit_events`, `audit_tool_events`, `audit_exports`
- `SELECT`, `INSERT`, and controlled column-level `UPDATE` on `audit_security_findings`
- no `DROP`, `ALTER`, `TRUNCATE`, `TRIGGER`, `SUPER`, `FILE`, `PROCESS`, or similar administrative privileges

Migration account:

- used only during setup or upgrades
- has DDL privileges
- must not be present in the long-running application environment
- on MySQL deployments with binary logging enabled, the DBA must either create triggers with a sufficiently privileged migration account or enable `log_bin_trust_function_creators=1` during the audited migration window

Startup/security baseline checks should fail if the runtime account has forbidden privileges.

## Retention

V1 documents retention and does not automatically delete audit records.

Defaults:

- Audit ledger: 7 years
- Login and privilege audit: 5 years
- Tool execution details: 2 years
- System health events: 1 year
- Audit export files: short-lived, deployment configurable

V2 should add monthly partitioning and archival to S3-compatible object storage. V1 health checks show row count, oldest event time, and newest event time to expose continuity gaps.

## Sync, Async, DLQ

`recordAuditEvent` supports `mode: "sync" | "async" | "auto"`.

Fail-close is based on an explicit action allowlist, not severity.

Fail-close actions include:

- `audit.export.requested`
- `audit.export.created`
- `audit.export.downloaded`
- `admin.user.role_changed`
- `admin.user.access_changed`
- `admin.user.password_reset`
- `tenant.created`
- `tenant.deleted`
- `file.downloaded`
- `skill.market.approved`
- `config.security_critical_changed`

For fail-close actions, audit must be written before the original operation runs. If audit cannot be persisted and DLQ cannot preserve the event, the original operation fails.

For non-fail-close actions, audit failure writes to local DLQ and the main flow continues.

DLQ:

- local JSONL files under `data/audit-dlq/`
- health page exposes queue size, last write failure, last drain time, and directory usage
- high disk usage raises health warnings
- v1 roadmap: AES-256-GCM encryption for DLQ and export files

## Export Flow

Audit export is a privileged operation and cannot bypass audit.

Flow:

1. Write `audit.export.requested` synchronously.
2. Generate export file.
3. Store with `storage_key`, `file_hash`, `file_size_bytes`, `format`, and expiry.
4. Write `audit.export.completed`.
5. Download writes `audit.export.downloaded` before streaming.

If step 1 fails, export is blocked.

## Redaction

Every metadata payload runs through:

- `redactSecrets`
- `redactPII`
- metadata truncation

Secret keys include password, token, secret, apiKey, cookie, authorization, credential, privateKey, gatewayToken, and botToken.

PII defaults include national ID patterns, bank cards, phone numbers, and email addresses. Rules must be deployment-configurable.

Do not put full prompts, file bodies, stdout/stderr, or LLM responses into metadata. Store hashes, byte counts, short previews, and references instead.

## Metadata Guidelines

Do not store:

- full prompt text
- full file content
- full stdout/stderr
- full LLM response
- secrets or raw tokens

Prefer:

- IDs and hashes
- byte lengths
- first 200 characters of safe previews
- error codes and policy codes
- detail `event_id` references

Metadata is limited to 16 KB. Oversized metadata is truncated and marked with `metadata_truncated` and `metadata_original_bytes`.

## Cross-Region Strategy

V1 treats the entry boundary as the audit source of truth.

- Shanghai entrypoint records user-visible boundary events.
- Singapore internal worker details are not mandatory in v1.
- V2 can forward Singapore detail events back to Shanghai asynchronously using `correlation_id`.

## Batch Operations

Batch operations write:

1. one parent event such as `agent.batch_deleted`
2. one child event per item such as `agent.deleted`

Child events use `correlation_id = parent.event_id`.

## Legacy Tables

Legacy tables remain in place but are not the new primary audit source:

- `claw_adoption_events`
- `tool_execution_audits`
- `business_agent_audit`
- `security_logs`
- `ip_access_logs`
- `agent_call_logs`
- `lx_coop_events`

V1 does not migrate or union legacy data. New admin audit pages read only the new ledger.

6. Add Security Audit admin tab and export flow.
7. Wire tool execution and browser policy events.

## V1 Acceptance

- Fresh install creates all 4 tables.
- WORM triggers reject UPDATE/DELETE where expected.
- Controlled updates on security findings obey the transition graph.
- Security health shows table presence, row count, oldest/newest event time, DLQ size, and recent audit failures.
- Login, password reset, model switch, skill install/review, Weixin bind/unbind, and agent lifecycle events are recorded.
- Metadata contains no obvious secrets or PII.
- Fail-close actions write audit before operation.
- Export cannot proceed when audit is unavailable.
