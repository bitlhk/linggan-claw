# 企业级 AI Agent 平台部署指南

> 面向金融机构 IT 架构师与技术负责人

---

## 一、企业 Agent 与个人 Agent 的本质区别

个人 Agent（如开源 OpenClaw）是一个"聪明的终端"——一个用户、一套配置、一个工作区。

企业 Agent 是一个"受控的组织能力"——多用户、多角色、多审批流，运行在合规框架内。

| 维度 | 个人 Agent | 企业 Agent 平台 |
|------|-----------|----------------|
| 用户模型 | 单用户 | 多租户，角色权限 |
| 会话隔离 | 单 workspace | 用户间物理隔离 |
| 工具执行 | 本地沙箱 | 受控沙箱 + 策略引擎 |
| 数据流转 | 无限制 | 审计 + 合规拦截 |
| 通知触达 | 终端输出 | 多渠道（微信/企微/飞书） |
| 定时任务 | 本地 cron | 平台级调度 + 投递 |
| 品牌定制 | 无 | 一套 Token、一键换肤 |
| 可观测性 | 日志 | 全链路审计 + 操作回放 |

**核心判断**：企业不需要更强的 LLM，需要的是 LLM 之上的**治理层**。

## 二、企业 Agent 平台的理想架构

```
┌────────────────────────────────────────────────────────────────┐
│  用户触达层                                                     │
│  Web UI / 微信 / 企业微信 / 飞书 / API                          │
└───────────────────────────┬────────────────────────────────────┘
                            │
┌───────────────────────────▼────────────────────────────────────┐
│  平台路由层（Platform Intent Engine）                            │
│                                                                │
│  消息进来 → 意图分类 → 路由决策                                  │
│    → AI 对话？     → Agent Runtime                              │
│    → 平台操作？    → 平台执行器（定时/通知/协作）                 │
│    → 专业任务？    → 推荐专业 Agent（PPT/代码/金融分析）          │
│                                                                │
│  策略引擎：auto / confirm / review（按意图类型可配置）           │
└───────────────────────────┬────────────────────────────────────┘
                            │
┌───────────────────────────▼────────────────────────────────────┐
│  Agent Runtime（LLM + 工具执行）                                │
│                                                                │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐      │
│  │ 对话推理  │  │ 代码沙箱  │  │ Web 搜索  │  │ 记忆系统  │      │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘      │
│                                                                │
│  工具路由器（Tool Router）                                       │
│    tool_call → 策略校验 → 沙箱/原生/拒绝                        │
└───────────────────────────┬────────────────────────────────────┘
                            │
┌───────────────────────────▼────────────────────────────────────┐
│  多租户隔离层（Tenant Isolation Layer）                          │
│                                                                │
│  (userId, agentId) → tenantToken (HMAC-SHA256, 不可逆)         │
│  每个租户独立 workspace / session / 审计日志                     │
│  租户间零共享：文件、内存、会话完全隔离                           │
└───────────────────────────┬────────────────────────────────────┘
                            │
┌───────────────────────────▼────────────────────────────────────┐
│  基础设施层                                                     │
│  模型网关 / 数据库 / 对象存储 / 容器运行时                       │
└────────────────────────────────────────────────────────────────┘
```

**关键设计决策**：

- **平台层和 Runtime 层职责分离**：Runtime 只管推理和执行，平台管用户、渠道、调度、合规。两层之间通过意图路由桥接，不耦合。
- **Agent Runtime 不感知用户身份**：它只知道"一个 session 在一个 workspace 里"，不知道"张三在用微信"。用户上下文由平台层持有。
- **专业 Agent 独立会话**：PPT 生成、代码执行、金融分析各自独立上下文，不污染主对话。平台层做导航推荐，不做代理转发。

## 三、安全架构：五层纵深防御

### 第一层：代码执行沙箱

所有用户触发的代码执行在 Docker 容器内运行：

| 隔离措施 | 说明 |
|---------|------|
| `--network=none` | 零网络访问 |
| `--read-only` | 根文件系统只读 |
| `--no-new-privileges` | 禁止提权 |
| seccomp profile | 系统调用白名单 |
| CPU/内存/进程数限制 | 资源封顶 |
| 5 分钟执行超时 | 自动中断 |
| 临时挂载 `/output` | 唯一可写目录，执行后清理 |

### 第二层：工具策略引擎（Tool Policy）

LLM 发出 tool_call 后，不直接执行，先过策略引擎：

```
tool_call 进来
  → 工具名映射（name → handler）
  → 命令黑名单检查（rm -rf / DROP TABLE / 凭证读取等）
  → 工作目录白名单校验
  → 环境变量过滤
  → 参数 schema 校验
  → 策略裁决：sandbox_exec / native / deny
```

被拒绝的调用返回结构化拒绝原因，LLM 可基于此调整策略。

### 第三层：平台级系统提示注入

每次请求在 LLM 的 system prompt 最前面注入平台安全规则（不可被用户指令或 Agent 配置覆盖）：

- 禁止读取/泄露凭证、API Key、.env 文件
- 禁止暴露基础设施信息（IP、端口、部署路径）
- 破坏性操作必须用户二次确认
- 三级审批规则：自动放行（只读）/ 单次确认（写入）/ 强制确认（破坏性）

### 第四层：多租户隔离（TIL）

| 隔离维度 | 实现 |
|---------|------|
| 身份脱敏 | (userId, agentId) → HMAC-SHA256 tenantToken，下游不可反推 |
| 物理隔离 | 每个租户独立目录，不共享文件系统 |
| 会话隔离 | 独立 sessionKey，不同用户的对话零交叉 |
| Agent 隔离 | 同一用户不同 Agent 也互相隔离 |

### 第五层：全链路审计

每个工具调用生成完整审计记录：

```json
{
  "auditId": "aud_1776000000_abc1234",
  "userId": 42,
  "agentId": "trial_xxx",
  "toolName": "exec",
  "command": "python3 analysis.py",
  "policyDecision": "allow",
  "executor": "sandbox",
  "exitCode": 0,
  "durationMs": 3200,
  "truncated": false,
  "createdAt": 1776000000000
}
```

支持按用户/Agent/时间/工具类型查询，满足金融机构操作可追溯要求。

## 四、企业必备功能特性

### 4.1 多渠道触达

企业用户不只在 Web 端操作，需要在日常工作工具中触达 Agent：

| 渠道 | 能力 | 实现方式 |
|------|------|---------|
| Web UI | 全功能交互 | WebSocket + HTTP SSE 双通道 |
| 微信 | 文字 + 语音消息 | iLink Bot API + 语音转文字 |
| 企业微信 | 通知推送 | Webhook |
| 飞书 | 通知推送 | Webhook |
| API/Webhook | 系统集成 | RESTful API |

关键设计：**Agent Runtime 不感知渠道**。渠道管理在平台层，Agent 只负责生成内容，平台负责投递。新增渠道不需要改 Agent。

### 4.2 平台意图路由

用户的自然语言不只是"聊天"，还包含平台操作意图：

- "每天早上 9 点查股价发到我微信" → 创建定时任务 + 配置渠道投递
- "删除那个定时任务" → 任务管理
- "帮我做个路演 PPT" → 推荐专业 Agent

意图识别采用两级架构：
- **L1 关键词打分**（本地，0ms）：过滤 99% 普通聊天
- **L2 轻量 LLM 分类**（仅 L1 命中时调用，<1s）：结构化提取参数

策略可配置：某些操作自动执行，某些需要用户确认，某些需要管理员审批。

### 4.3 定时任务与自动化

Agent 不只是被动应答，还能主动执行：

- 支持 cron 表达式、固定时间、间隔执行
- 执行结果自动投递到用户指定渠道
- 每次执行记录 token 消耗、耗时、状态
- 最小间隔限制（防滥用）、每用户任务数上限

### 4.4 专业 Agent 协作

不同业务场景由专业 Agent 处理，各自独立上下文：

- 主对话 Agent → 日常聊天、信息查询
- PPT Agent → 文档生成
- 代码 Agent → 沙箱内代码执行
- 金融分析 Agent → 专业投研

平台层负责智能导航（推荐合适的 Agent），不负责跨 Agent 数据传递（避免数据泄露）。

### 4.5 品牌定制化

企业部署需要白标能力：

| 定制项 | 实现 |
|--------|------|
| 品牌色 | CSS Token 体系，一个值驱动全站配色 |
| Logo / Favicon | 配置文件替换 |
| 产品名称 / 标语 | 数据库配置，运行时生效 |
| AI 身份 / 人设 | System Prompt + SOUL.md |
| 字体 / 字号比例 | Token 体系支持（可选） |

设计原则：**品牌维度（accent color）和主题维度（dark/light）正交**。工行用红色 accent 可以配暗色主题，也可以配浅色主题。

## 五、审计与合规要求清单

面向金融机构，以下审计能力是必需的：

### 5.1 操作审计

| 审计项 | 记录内容 |
|--------|---------|
| 用户登录 | 时间、IP、设备、认证方式 |
| 对话记录 | 每轮 user/assistant 消息（可配置保留策略） |
| 工具调用 | 工具名、参数、执行结果、耗时、策略决策 |
| 文件操作 | 上传/下载/生成，文件名、大小、路径 |
| 定时任务 | 创建/修改/删除/执行，任务参数、执行结果 |
| 配置变更 | Agent 设置、品牌配置、权限变更 |

### 5.2 安全审计

| 审计项 | 记录内容 |
|--------|---------|
| 策略拦截 | 被拒绝的工具调用，拒绝原因，原始命令 |
| 异常检测 | IP 黑名单触发、4xx 频率异常、可疑请求模式 |
| 提权尝试 | 尝试读取凭证、尝试绕过沙箱、尝试网络外联 |

### 5.3 合规要求

| 要求 | 实现 |
|------|------|
| 操作可追溯 | 每个 tool_call 有唯一 auditId |
| 数据不出域 | 沙箱无网络、租户间物理隔离 |
| 可审计 | 结构化日志，支持按条件查询 |
| 可回放 | 保留完整对话上下文 + 工具执行链 |
| 权限最小化 | 沙箱只读、seccomp 白名单、环境变量过滤 |

## 五点五、Agent Team：从单兵到团队协作

### 为什么需要 Agent Team

单个 Agent 能力有限——一个通才 Agent 在专业场景下表现不如专家 Agent，而一个专家 Agent 又无法覆盖日常对话。企业需要的是一支 **Agent 团队**：有主帅、有专家、有工匠，各司其职、协同工作。

### 团队架构：三层 Agent 体系

```
┌─────────────────────────────────────────────────────┐
│  主对话 Agent（个人助手）                              │
│  角色：日常对话、信息查询、任务分发                     │
│  能力：Web 搜索、代码执行、记忆管理                    │
│  定位：用户的"入口"和"管家"                           │
└─────────────────────┬───────────────────────────────┘
                      │ 平台意图路由（推荐/导航）
          ┌───────────┼───────────┐
          ▼           ▼           ▼
┌─────────────┐ ┌──────────┐ ┌──────────────┐
│  专家 Agent  │ │ 专家Agent │ │  工匠 Agent   │
│  金融分析组   │ │ 决策支持组│ │  生产力工具组  │
│             │ │          │ │              │
│ · 股票分析   │ │ · 信贷风控│ │ · PPT 生成    │
│ · 债券投研   │ │ · 理赔决策│ │ · 代码助手    │
│ · 交易策略   │ │ · 理财规划│ │ · 幻灯片制作  │
│ · 共享智脑   │ │          │ │ · 技能开发    │
└─────────────┘ └──────────┘ └──────────────┘
```

### 团队协作模式

**模式一：导航式协作（已实现）**

用户在主对话中表达意图，平台识别后推荐合适的专家 Agent：

```
用户："帮我做个路演 PPT"
  → 平台识别 PPT 意图
  → 推荐卡片："检测到 PPT 需求，推荐使用 PPT 助手"
  → 用户选择打开 → 专家 Agent 接管
  → 独立会话，不污染主对话上下文
```

优势：零上下文污染、用户有选择权、实现简单可靠

**模式二：多 Agent 流水线（已在金融分析中实现）**

复杂任务由多个专业 Agent 流水线处理：

```
用户："分析工商银行投资价值"
  → 交易策略 Agent 启动
  → 基本面分析节点 → 产出基本面报告
  → 技术面分析节点 → 产出技术指标
  → 情绪面分析节点 → 产出市场情绪
  → 新闻面分析节点 → 产出资讯摘要
  → 投资辩论节点   → 多空博弈
  → 最终决策：OVERWEIGHT / HOLD / UNDERWEIGHT
```

四个分析节点并行执行，辩论节点综合裁决。每个节点可以使用不同的数据源和模型。

**模式三：跨 Agent 协作请求（已实现协议层）**

Agent A 可以向 Agent B 发起协作请求：

```
用户对 Agent A："帮我查个数据然后做成 PPT"
  → Agent A 完成数据查询
  → 向 PPT Agent 发起协作请求
  → 请求进入审批队列（可配置自动/人工审批）
  → PPT Agent 接收任务并执行
  → 结果回传给 Agent A
```

协作协议支持：请求 → 审批 → 执行 → 回传 → 审计，全链路可追溯。

### 团队管理：企业级约束

| 维度 | 个人场景 | 企业场景 |
|------|---------|---------|
| Agent 可见性 | 所有 Agent 可见 | 按角色/部门分配可见 Agent |
| 调用权限 | 自由调用 | 策略控制（auto/confirm/review） |
| 数据流转 | 无限制 | 跨 Agent 传数据需审批 |
| 资源消耗 | 无限制 | 按 Agent 计量 token 消耗 |
| 会话隔离 | 各自独立 | 各自独立 + 租户级隔离 |

### 共享智脑：团队级知识融合

除了专家分工，企业还需要一个"团队大脑"——融合多个专业领域知识的共享 Agent：

```
共享智脑 Agent
  ├── 信贷风控框架（银保监指引）
  ├── 债券投研框架（中债估值方法论）
  ├── 保险精算框架（再保协会标准）
  ├── 合规审查框架（监管要求）
  ├── ...
  └── 支持跨领域组合问答
```

用户提出跨领域问题（如"贷款 500 万买厂房需要哪些保险保障"），共享智脑能横跨信贷、保险、风控多个框架给出综合回答。

**关键设计**：共享智脑是**团队级共享**（同一机构所有用户共用知识框架），不是用户级私有的。隔离边界画在机构（tenant）而不是个人（user）。

### 演进路线

```
Phase 1（已实现）：导航式协作
  → 主对话推荐专家 Agent，用户手动切换

Phase 2（已实现）：多 Agent 流水线
  → 复杂任务自动编排多个分析节点

Phase 3（协议就绪）：跨 Agent 协作
  → Agent 间请求-审批-执行-回传

Phase 4（规划中）：智能编排
  → 平台根据任务复杂度自动选择单 Agent / 多 Agent / 流水线模式
  → 资源预估 + 审批前置
```

## 五点六、多智能体协调模式：五种主流范式全覆盖

业界公认的多智能体协调有五种主流模式（参考 Anthropic 2026 年发布的《Multi-Agent Coordination Patterns》）。企业 Agent 平台应根据场景灵活选用，而非只依赖单一模式。

### 模式一：调度-子智能体（Orchestrator-Subagent）— 核心架构

**运作原理**：中央调度者（Intent Agent）接收用户消息，分析意图后将任务分派给专业子智能体处理。

**实现**：

```
用户消息 → Intent Agent（L1 关键词打分 + L2 LLM 分类）
    │
    ├─ 普通对话 → 主对话 Agent
    ├─ 定时任务 → 平台调度器
    ├─ PPT 制作 → 幻灯片 Agent
    ├─ 信贷评估 → 智贷决策 Agent
    └─ 股票分析 → 交易分析 Agent
```

**关键设计**：
- 两级筛选：L1 关键词打分（0ms，过滤 97% 普通聊天）+ L2 LLM 分类（仅 3% 命中时调用）
- 子智能体列表从数据库动态读取，新增 Agent 无需改代码
- 三级策略：自动执行 / 用户确认 / 管理员审批

### 模式二：生成-验证者（Generator-Verifier）— 合规场景

**运作原理**：Agent 生成内容后，独立的验证器按标准逐项检查，确保输出质量。

**实现**：信贷风控场景

```
智贷决策 Agent 生成信贷报告（流式输出）
    → 报告生成完毕
    → 合规验证器逐项检查（10 项，基于金融监管总局现行法规）
    → 追加合规自检卡片
```

**验证清单示例**：

| 检查项 | 依据 |
|--------|------|
| 五级分类是否给出 | 《商业银行金融资产风险分类办法》2023 |
| 还款来源分析 | 《流动资金贷款管理办法》2024 修订版 |
| 免责声明 | 监管合规要求 |
| 红线检查 | 不预测盈利/不承诺担保/不做准入决策 |

**关键设计**：流式输出不中断（保持用户体感），验证在流结束后追加，用户同时看到报告和质检结果。

### 模式三：智能体团队（Agent Teams）— 复杂分析

**运作原理**：多个专业 Agent 长期积累领域知识，并行处理独立子任务，最终汇总。

**实现**：智能交易分析

```
用户："分析工商银行投资价值"
    → 基本面分析节点（财务报表、估值水平）   ─┐
    → 技术面分析节点（K线、MACD、RSI）      ─┤ 并行执行
    → 情绪面分析节点（市场情绪、资金流向）   ─┤
    → 新闻面分析节点（公告、政策、舆情）     ─┘
    → 投资辩论节点（多空博弈）
    → 最终决策：OVERWEIGHT / HOLD / UNDERWEIGHT
```

**关键设计**：四个节点各自积累领域知识，越用越专业。辩论节点综合多维度数据后裁决，避免单一视角偏差。

### 模式四：消息总线（Message Bus）— Agent 间协作

**运作原理**：Agent 通过发布-订阅机制通信，路由器按 requestId 精准分发，新 Agent 只需订阅即可接入。

**实现**：跨助手协作协议

```
Agent A（用户甲）
    → 发起协作请求（发布事件）
    → 平台协作总线（collabStreamMap 路由）
    → Agent B（用户乙）接收并执行
    → 实时流式结果回传给 Agent A
    → 完成通知推送
```

**关键设计**：
- 松耦合：发起方不需要知道目标 Agent 的实现细节
- 审批机制：协作请求需目标方批准后才执行（企业级管控）
- 实时流：执行过程中结果实时流式回传，不需要等全部完成
- 全链路审计：请求 → 审批 → 执行 → 回传，每步可追溯

### 模式五：共享状态（Shared State）— 跨领域知识融合

**运作原理**：无中心指挥官，多个 Agent 共享持久化知识基座，各自读取、处理、写回，形成知识演进。

**实现**：共享智脑（8 大金融框架融合）

```
用户："贷款 500 万买厂房需要什么保险保障？"
    → 信贷框架：分析抵押物评估、还款能力
    → 保险框架：读取信贷分析 → 补充财产险/责任险配置
    → 风控框架：读取两者 → 综合风险评估和缓释建议
    → 输出：跨领域综合方案
```

**关键设计**：
- 8 个专业框架共享同一个知识基座，互相引用对方的分析结果
- 共享层面在机构（tenant）而非个人（user），同一机构所有用户共用知识框架
- 设有终止条件，避免无限循环消耗算力

### 模式对比与选择建议

| 模式 | 适用场景 | 核心优势 |
|------|---------|---------|
| 调度-子智能体 | 日常使用，任务分发 | 最广泛，建议作为默认架构 |
| 生成-验证者 | 合规场景，质量关键 | 输出可信度高 |
| 智能体团队 | 复杂分析，多维度 | 并行+专业积累 |
| 消息总线 | 跨用户/跨团队协作 | 松耦合，易扩展 |
| 共享状态 | 跨领域知识融合 | 知识互相踩肩膀 |

**实践建议**：从调度-子智能体开始，观察系统在哪里卡了脖子，然后根据痛点演进到其他模式。五种模式是积木，生产环境通常混搭使用。

## 六、部署架构建议

### 最小部署（PoC / 试点）

```
单节点部署：
  Agent Runtime（Gateway）     → 1 进程
  平台服务（Node.js）          → 1 进程
  数据库（PostgreSQL）         → 1 实例
  容器运行时（Docker/Podman）   → 沙箱执行
```

适合 10-50 用户试点，单台 8C16G 服务器即可。

### 生产部署

```
负载均衡（Nginx/SLB）
  │
  ├─ 平台服务 ×2（无状态，水平扩展）
  │
  ├─ Agent Runtime ×N（按用户数扩展）
  │
  ├─ 数据库（主从）
  │
  └─ 沙箱节点池（独立节点，安全隔离）
```

### 私有化部署要点

| 要点 | 建议 |
|------|------|
| 模型选择 | 支持对接私有化大模型（GLM/DeepSeek/通义等） |
| 网络隔离 | Agent Runtime 与沙箱节点网络隔离 |
| 数据存储 | 所有数据落本地，不出内网 |
| 证书管理 | 内网 HTTPS，自签证书或企业 CA |
| 监控告警 | 接入企业监控体系（Prometheus/Grafana） |

---

## 七、总结

企业级 AI Agent 平台不是"把开源 Agent 装在服务器上"，而是在 Agent Runtime 之上构建一整套**治理体系**：

1. **平台层管用户和连接**：身份、渠道、调度、合规
2. **Runtime 层管推理和执行**：LLM、工具、沙箱
3. **两层之间通过意图路由桥接**：不耦合、可独立演进

这套架构的核心价值不在 AI 本身，在于**让 AI 在企业的合规框架内持续、安全、可审计地工作**。
