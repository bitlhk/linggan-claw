# 灵虾多租户隔离架构设计

> 给大企业的 Agent 平台多租户隔离方案
> 作者：灵感 / 灵虾团队 · 2026-04-10

---

## 一、问题重述

```
用户 A (子虾 lgc-aaa) ┐
用户 B (子虾 lgc-bbb) ├─→ 同一个远程 claude-code 服务
用户 C (子虾 lgc-ccc) ┘        (3.16.70.167:19800)
```

三个用户都用同一个远程 agent，但他们的**对话、文件、身份、额度**必须互相看不见。

这是典型的**多租户隔离问题**（multi-tenancy isolation）—— 企业级 AI Agent 平台最核心的架构挑战之一。

---

## 二、隔离的五个维度

| 维度 | 必要性 | 当前状态 | 风险 |
|---|---|---|---|
| **① 对话上下文隔离** | 🔴 必需 | ✅ 有（按 userId 区分 sessionKey） | 低 |
| **② Workspace 文件隔离** | 🔴 必需 | ❌ **无** | **高** —— A 写的文件 B 能读 |
| **③ 身份信息隔离** | 🟡 强烈建议 | ⚠️ userId 明文出现在 sessionKey | 中 —— LLM 越狱可能泄露 userId |
| **④ 配额限流隔离** | 🟡 建议 | ⚠️ 只有全局限流 | 中 —— 一人把服务打爆 |
| **⑤ 审计隔离** | 🟢 合规要求 | ⚠️ 部分 | 合规风险 |

---

## 三、现状最严重的漏洞：Workspace 共享

```
3.16.70.167:19800 的 claude-code 服务
  └── workspace: /workspace/                ← 全局一个目录
        ├── 用户A 写的 icbc-analysis.html   ← 所有人都能看到
        ├── 用户B 写的 private-report.md
        └── 用户C 写的 confidential.json
```

这是典型的**未隔离**。如果 A 让 agent `ls /workspace`，能看到 B 和 C 的所有产物。

---

## 四、三种隔离模式对比（核心决策）

### 维度对比

| 维度 | 每用户独立实例 | 架构隔离（单实例多 tenant） |
|---|---|---|
| **数据泄露风险** | 极低 | 低（靠 workspace 隔离 + tool 白名单） |
| **资源成本** | ❌ 100x | ✅ 1x |
| **冷启动延迟** | ❌ 10-30s | ✅ 50ms |
| **协作能力** | ❌ 弱 | ✅ 强 |
| **运维复杂度** | ❌ 线性爆炸 | ✅ 固定 |
| **扩容平滑度** | ❌ 差 | ✅ 好 |
| **计费简单度** | ❌ 难 | ✅ 简单 |
| **进程崩溃影响** | ✅ 单点 | ⚠️ 影响多人 |
| **安全合规文档好写** | ✅ "天然隔离" | ⚠️ 要写清楚隔离措施 |
| **调试支持** | ❌ 隐私问题 | ✅ 统一日志 |
| **升级** | ❌ 难（要重启每个实例） | ✅ 重启一个服务 |

**唯一实例隔离胜出的**：安全合规文档好写（客户一听"每人独占进程"就放心）

### 实例隔离的十个坑（容易被忽视）

#### 1. 资源成本线性爆炸

```
100 用户 = 100 个 claude-code 进程
每个 500MB 内存 = 50GB 常驻
每个 2GB workspace = 200GB 磁盘
每个冷启动 10 秒
```

1000 用户就是 500GB 内存 + 2TB 磁盘，普通机器扛不住。

#### 2. 身份层仍然要做

实例隔离 ≠ 认证鉴权。你依然需要：

- **谁能启动新实例？** —— 不能让随便一个 HTTP 请求就启动一个实例（DDoS）
- **谁能访问某个实例？** —— 必须有 session→user 的映射，不然 A 能访问 B 的实例
- **实例怎么销毁？** —— 用户下线后实例是常驻还是 gc？gc 策略错了就"数据丢失"

这一层工作量**跟没做实例隔离差不多**。

#### 3. 跨实例的共享状态怎么办

企业场景里很多东西就是要共享：

| 共享需求 | 实例隔离下的难题 |
|---|---|
| 公司知识库 / RAG | 100 个实例各拉一份？成本爆炸 |
| 共享技能（skill 市场） | 每个实例独立装？还是共享磁盘？ |
| 团队文档库 | 跨实例访问控制很难做 |
| 全公司审计日志 | 日志分散在 100 个实例，汇总难 |
| 多人协作的同一项目 | 你在实例 A，同事在实例 B，怎么协作？ |
| 模型 API 限流 | 上游模型限流是全局的，你的实例隔离没用 |

**实例隔离破坏了"企业协作"的默认假设**。

#### 4. 冷启动延迟劣化体验

- **共享实例**：50ms 返回
- **独占实例**：10-30 秒冷启动（加载模型、初始化 tool、读 skill）

想缓解就得**常驻所有实例**，回到问题 1。

#### 5. 模型层面的隔离依然要做

就算进程隔离了，你还是要保证 LLM 不被用户：
- 越狱出系统 prompt
- 通过工具执行读到宿主机文件
- 通过 `curl` 访问内网资源

这些跟"实例隔离没隔离"毫无关系，需要独立的 prompt 加固、工具白名单、网络隔离。

#### 6. 运维复杂度指数级上升

```
100 用户 × 100 实例的运维开销：
  • 100 份日志聚合
  • 100 个健康检查
  • 100 套指标监控
  • 100 个版本升级
  • 100 个崩溃重启
  • 100 份备份
```

#### 7. 扩容不平滑

机器加不进去的时候怎么办？

- **共享服务**：加 2x 实例就能 2x 吞吐（无状态）
- **实例隔离**：加机器之后存量用户无法迁移（状态都在原机器），新用户才用新机器 → **负载不均**

除非加一个**实例调度器**把用户在机器间迁移，又是一层新复杂度。

#### 8. 计费困难

企业经常按部门/项目分账：
- **共享服务**：一个 metering 层统计每用户用量
- **实例隔离**：需要在每个实例里做 metering 再汇总，或者在调度层拦截

#### 9. "实例"本身也可以被逃逸

```
Docker 容器 ≠ 100% 安全
Linux namespace 有逃逸漏洞历史（CVE）
共享内核 = 共享攻击面
```

真要做"绝对隔离"得用 VM（KVM、Firecracker），成本又翻 10 倍。

#### 10. 调试/支持的不可及

用户报 bug："我在代码助手里做 XX，失败了"。

- **共享服务**：直接看统一日志
- **实例隔离**：SSH 进用户的实例，或者把实例快照 dump 回来，隐私合规问题来了

## 五、三层隔离架构（推荐做法）

### 层 1：灵虾侧（已有 + 要补）

```
灵虾后端
  ├─ sessionKey 按 userId 生成           ✅ 已有
  ├─ 传 tenantToken 而不是 userId         ❌ 要补
  ├─ 给远程请求注入 workspace 路径参数     ❌ 要补
  └─ 灵虾限流器按 user 限流               ✅ 已有
```

### 层 2：远程 claude-code 服务侧（要改）

```
远程 claude-code 服务
  ├─ 每个请求带 workspace header
  ├─ 根据 workspace 动态 chroot/mount
  ├─ 进程内按 session key 路由到不同子 context
  └─ tool 调用前加 user 上下文检查
```

### 层 3：文件系统层（物理隔离）

```
远程服务器:
  /data/tenants/
    ├── {tenantTokenA}/
    │     ├── workspace/   ← 用户 A 独占
    │     ├── logs/
    │     └── skills/
    ├── {tenantTokenB}/
    │     └── workspace/   ← 用户 B 独占
    └── {tenantTokenC}/
          └── workspace/
```

## 六、三种隔离强度（按场景选）

| 强度 | 做法 | 资源 | 安全 | 建议场景 |
|---|---|---|---|---|
| **弱隔离**（当前） | 共享进程 + sessionKey 区分 | 1x | ⚠️ 低 | 内部测试 |
| **中隔离**（推荐演进） | 共享进程 + 动态 workspace 路径 | 1.2x | ✅ 中高 | 普通企业客户 |
| **强隔离** | 每 tenant 一个独立进程 + 独立磁盘分区 | Nx | ✅ 极高 | 金融 / 政务 / 极敏感 |

对工行这种客户，**强隔离是必需**。但大部分情况**中隔离就够**。

## 七、给大企业建议的参考架构

```
             企业用户 1000 人
                    │
                    ▼
          ┌────────────────────┐
          │    灵虾平台前端     │
          └──────────┬─────────┘
                     │
                     ▼
          ┌────────────────────┐
          │   灵虾后端 + 鉴权   │
          │  • 用户认证         │
          │  • tenantToken 下发 │
          │  • 配额管理         │
          │  • 限流             │
          └──────────┬─────────┘
                     │ 每次请求带 tenantToken
                     │ 不暴露真实 userId
                     ▼
          ┌────────────────────┐
          │   业务 Agent 池     │
          ├────────────────────┤
          │  代码 Agent (共享)  │   ← 进程级别共享
          │  分析 Agent (共享)  │      但每请求隔离 workspace
          │  PPT  Agent (共享)  │
          └──────────┬─────────┘
                     │
                     ▼
          ┌────────────────────┐
          │   物理隔离存储      │
          │  /data/tenants/    │
          │    ├─ tenantA/     │
          │    ├─ tenantB/     │
          │    └─ tenantC/     │
          └────────────────────┘
```

### 关键设计点

1. **tenantToken 单向映射**
   - 灵虾后端维护 `userId ↔ tenantToken` 映射表
   - 远程 agent 只看到 tenantToken（一个 UUID），看不到真实用户身份

2. **Workspace 路径由灵虾后端注入**
   - 远程 agent 不能自己决定写哪
   - workspace 路径是灵虾传的 header

3. **进程共享 + 数据隔离**
   - 不需要给每个用户起一个 claude-code 进程（成本爆炸）
   - 同一个进程但每请求切 workspace 就够

4. **审计双向可追溯**
   - 灵虾后端记录"userA 调用了什么"
   - 远程服务记录"tenantX 执行了什么"
   - 两边通过 tenantToken 可以关联查
   - 远程服务本身无法反向推出是哪个真人

## 八、真正的企业级答案：分级隔离

大厂通常这么做（AWS / OpenAI / Anthropic 都是这个思路）：

```
普通用户（免费 / Plus）
  ↓
  共享 agent 服务
  workspace 隔离 + sessionKey 隔离
  成本低、体验好
  适用 95% 场景

高价值企业客户（专属部署）
  ↓
  独立 agent 进程
  完全物理隔离
  贵但省心
  适用 5% 场景

极敏感客户（银行 / 政务）
  ↓
  完全私有化部署
  自己的机房、自己的模型
  最贵、最安全
```

**默认共享服务 + workspace/session 级隔离，对专属客户卖"专属实例"作为付费选项。**

## 九、落地路线图

### 阶段一：现状（已完成 + 补）

- ✅ sessionKey 按 userId 生成（已有）
- ❌ Workspace 按 tenant 隔离（要补，核心缺失）
- ⚠️ userId 明文传递（要改成 tenantToken）

### 阶段二：3 步落地（共 4 天）

**Step 1（1 天）：灵虾后端传 workspace header**
- 灵虾后端在调远程 agent 时多传一个 header：
  ```
  x-tenant-workspace: /data/tenants/{hash(userId)}/workspace
  x-tenant-token: {随机 token，映射到 userId}
  ```

**Step 2（2 天）：远程 claude-code 服务改造**
- 接收 header 里的 workspace 路径
- 启动对应 session 时传给 `sessions.create` 的 `workspace` 参数
- 确保 tool 调用（bash/read/edit）受限在这个目录下

**Step 3（1 天）：物理隔离 + 目录自动创建**
- 远程服务器上预留 `/data/tenants/` 根目录
- 首次请求时自动 `mkdir -p` 对应 tenant 目录
- 配额：每个 tenant 目录磁盘空间限制（quota）

### 阶段三：未来演进

| 时机 | 做什么 |
|---|---|
| **现在** | 共享模式 + workspace 隔离（3 天能做完） |
| **拿下第一个金融客户** | 给他们专属实例 + 专属目录挂载 |
| **客户量 > 10** | 做一个"实例调度层"，按 agent 类型决定共享还是专属 |
| **客户量 > 100** | 上 K8s，按 label 调度，共享/专属无缝切换 |

## 十、给大企业汇报的核心话术

> **"多租户隔离不是 0 和 1 的问题，而是一个谱系：从最弱的 prompt 约束，到中等的 workspace 隔离，再到最强的进程级隔离。企业应根据自己的数据敏感度和合规要求，在成本和隔离强度之间做取舍。**
>
> **灵虾的设计理念是'通过 tenant token 解耦身份与执行' —— 灵虾负责认证和授权，业务 agent 不需要知道真实用户是谁，只需要按 tenant token 做执行隔离。这样单个 agent 服务可以服务所有用户，同时数据物理隔离，管理成本低、安全性高。**
>
> **我们提供两种部署模式：**
>
> - **共享模式（多租户）**：成本低、协作友好、运维简单。通过 sessionKey + workspace 物理隔离保证数据不串，适合绝大多数场景。
> - **专属模式（单租户）**：每个业务独立进程、独立存储，提供最高级别的物理隔离，适合核心业务或合规要求极高的场景。
>
> **两种模式可以在同一平台内并存** —— 普通能力走共享，核心业务走专属。"

## 十一、结论

**灵虾现阶段不要做实例隔离**。理由：

1. 用户量还不到那个级别
2. 共享模式 + workspace 隔离已经能满足 90% 的安全需求
3. 实例隔离的坏处在现阶段全部会命中（成本、运维、协作）

**最关键的一件事**：**现在把共享模式的 workspace 隔离做扎实。** 这才是地基。以后要上专属实例也是水到渠成的升级，不是推倒重来。

**"过度工程是创业最大的坑。"**

## 附录：关键名词解释

| 术语 | 含义 |
|---|---|
| **Tenant** | 租户 —— 一个独立的使用单位，可以是一个用户、一个部门、一个公司 |
| **tenantToken** | 租户令牌 —— 代替真实 userId 传给下游服务的匿名化 ID |
| **Workspace** | 工作区 —— agent 能读写的文件目录 |
| **Session Key** | 会话 key —— OpenClaw 里标识一个对话实例的唯一 ID |
| **Multi-tenancy** | 多租户 —— 一个系统同时服务多个独立租户的架构 |
| **Chroot** | 改变根目录 —— Linux 文件系统隔离技术 |
| **Quota** | 配额 —— 磁盘/CPU/内存的硬限制 |
